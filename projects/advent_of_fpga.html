Background

I took on Jane Street’s Advent of FPGA challenge, which reimagines Advent of Code problems as hardware designs. The goal is to solve problems using FPGA-oriented thinking rather than traditional software.

This was my introduction to both OCaml and Hardcaml. Coming from a hardware background, the functional programming model was initially unfamiliar. I worked through OCaml examples and existing Hardcaml designs, and used Jane Street’s template to set up a structured hardware project.

The repository includes both a Hardcaml implementation of the hardware design and a Go reference model, which I used as a golden model for verification.

Solution Approach

(* Part 1 — Instantaneous rotation *)
let rotate pos dir amount =
  match dir with
  | 'L' -> if pos - amount < 0 then pos + 100 - amount else pos - amount
  | 'R' -> if pos + amount > 99 then pos - 100 + amount else pos + amount
  | _ -> pos


(* Part 2 — Click-by-click rotation *)
let clicking_rotate pos dir amount =
  let step (pos, count) _ =
    let pos' =
      match dir with
      | 'L' -> if pos = 0 then 99 else pos - 1
      | 'R' -> if pos = 99 then 0 else pos + 1
      | _ -> pos
    in
    let count' = if pos' = 0 then count + 1 else count
    in
    (pos', count')
  in
  let final_pos, clicks =
    Seq.(ints 0 |> take amount |> fold_left step (pos, 0))
  in
  (final_pos, clicks)

The OCaml reference model implements both parts of the problem. The first computes the final position using modular arithmetic, while the second models the rotation as a sequence of single-step updates, counting each time the dial passes through zero.

What I Learned

This project changed how I think about problem solving. Instead of writing a sequence of steps, I had to describe a system that continuously processes data, where every cycle matters and state evolves over time.

I learned how to translate a high-level problem into a streaming datapath by breaking it into registers, combinational logic, and clear state transitions. Even simple problems required careful thought about how values move and update on each clock edge.

It also reinforced the importance of timing and correctness at the cycle level. Designing the logic is only part of the problem. Ensuring that signals arrive at the right time and that edge cases are handled across cycles is just as important.

Overall, this experience strengthened my intuition for thinking in hardware, designing systems defined by structure and timing rather than control flow.
